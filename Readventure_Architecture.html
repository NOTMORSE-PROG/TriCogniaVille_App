1. Project Overview

Readventure is a village-restoration reading game where Grade 7 students take the role of a "Readventurer." The village starts mostly silent — buildings appear grayscale and locked. By completing reading missions (decoding, fluency, comprehension), students unlock buildings, which animate from gray to vibrant color, restoring the village.

Core Mechanic

Locked items = grayscale. Complete quest → building animates to full color. Visual reward tied to learning.

Reading Skills

Decoding (word recognition) → Fluency (smooth reading) → Comprehension (meaning)

Adaptive Engine

4 reading levels (non-reader → independent). Same village, different difficulty per student.

3. System Architecture

3.1 Overall Data Flow

[ Android Device — Godot 4 App ] │ ├─ godot-sqlite ← offline source of truth (always writes here first) │ └─ HTTPRequest node → POST /sync (JSON) │ [ Cloudflare Worker ] ← free, 100K req/day │ DB credentials stored as Worker secrets │ Uses @neondatabase/serverless HTTP driver │ [ Neon PostgreSQL ] ← free, 0.5 GB, auto-suspend 5 min │ [ Cloudflare Pages — Teacher Dashboard ] HTML/JS → Cloudflare Worker → Neon

3.2 Offline-First Strategy

The app works 100% offline. SQLite on-device handles all quest saves. When WiFi is detected, Godot posts a batch of pending quest results to the Cloudflare Worker. The Worker upserts data into Neon (idempotent — safe to retry). The app then marks local records as synced = 1. This handles spotty school WiFi gracefully.

4. Grayscale → Color Unlock Mechanic

4.1 Shader (canvas_item, per-sprite)

// res://shaders/building_unlock.gdshader
shader_type canvas_item;
uniform float color_amount : hint_range(0.0, 1.0) = 0.0; // 0=gray, 1=full color

void fragment() {
    vec4 tex = texture(TEXTURE, UV);
    float luma = dot(tex.rgb, vec3(0.299, 0.587, 0.114)); // perceptual luminance
    COLOR.rgb = mix(vec3(luma), tex.rgb, color_amount);   // blend gray ↔ color
    COLOR.a = tex.a;
}

4.2 Unlock Animation (GDScript)

func unlock_building() -> void:
    is_unlocked = true
    $LockIcon.visible = false
    # Smooth 1.5s tween: gray (0.0) → full color (1.0)
    var tween = create_tween().set_ease(Tween.EASE_IN_OUT).set_trans(Tween.TRANS_CUBIC)
    tween.tween_method(
        func(v): $Sprite2D.material.set_shader_parameter("color_amount", v),
        0.0, 1.0, 1.5
    )
    tween.tween_callback(func(): $CPUParticles2D.emitting = true)
    GameManager.record_unlock(building_id)
⚠️ Critical: Each building must call ShaderMaterial.new() in _ready() to get its own material instance. Sharing one material resource causes all buildings to go gray or color together.

Responsive Rules

  • All HUD elements use Control node anchors (top-left, bottom-right etc.) — pin to screen edges
  • Quest panel is a centered Panel with max-width 1200px — works on phones and tablets
  • Building positions stored as viewport-relative ratios, not fixed pixels
  • Font sizes scale at runtime: base_size × (viewport_width / 1920.0)
  • Safe area padding via DisplayServer.get_display_safe_area() for notched phones

6. Quest System

Type 1 — Multiple Choice

4 options. Tests Comprehension. Correct answer highlights green. Used for inference and main idea questions.

Type 2 — Fill-in-the-Blank

LineEdit triggers soft keyboard. Tests Decoding. Answer input placed in upper screen to avoid keyboard overlap.

Type 3 — Drag-and-Drop

Word chip rearrangement. Tests Fluency / sentence structure. Uses Godot's built-in drag-and-drop API.

Quest Content Format (JSON)

{
  "building_id": "library",
  "quests_by_level": {
    "1": { "task_type": "multiple_choice", "passage": "...", "question": "...", "options": ["A","B","C","D"], "correct": 0 },
    "2": { "task_type": "fill_blank",      "passage": "...", "sentence": "The ___ sat by the well.", "answer": "cat" },
    "4": { "task_type": "drag_drop",       "passage": "...", "words": ["The","cat","sat","by","the","well"] }
  }
}

Adaptive Difficulty

Moving window of last 5 quests: ≥80% pass rate → level up. ≤40% → level down. Same building, different JSON level key.

7.2 Neon PostgreSQL Schema (Phase 2)

CREATE TABLE students (
  id TEXT PRIMARY KEY, name TEXT, reading_level INT DEFAULT 1, xp INT DEFAULT 0
);
CREATE TABLE quest_attempts (
  id SERIAL PRIMARY KEY, student_id TEXT REFERENCES students(id),
  quest_id TEXT, building_id TEXT, passed BOOLEAN, attempts INT, completed_at TIMESTAMPTZ
);
CREATE TABLE building_states (
  student_id TEXT REFERENCES students(id), building_id TEXT,
  unlocked BOOLEAN DEFAULT FALSE, unlocked_at TIMESTAMPTZ,
  PRIMARY KEY (student_id, building_id)
);

7.3 Cloudflare Worker (API Middleware)

// worker.js — Cloudflare Worker (free: 100K req/day)
import { neon } from '@neondatabase/serverless';

export default {
  async fetch(request, env) {
    const sql = neon(env.DATABASE_URL);  // secret — never in APK
    const { student_id, quest_results } = await request.json();
    for (const q of quest_results) {
      await sql`INSERT INTO quest_attempts
        (student_id, quest_id, building_id, passed, attempts, completed_at)
        VALUES (${student_id}, ${q.quest_id}, ${q.building_id},
                ${q.passed}, ${q.attempts}, NOW())`;
    }
    return Response.json({ ok: true });
  }
};

9. Implementation Phases

Phase 0 — Visual Design (1–2 weeks) ← START HERE

  • Import Kenney.nl assets, set up village map layout
  • Full-color vibrant background — sky, terrain, decorations
  • Apply grayscale shader to locked buildings, padlock icons
  • Tune color palette for maximum gray-vs-color contrast
  • Animate one sample unlock (gray → color + sparkles) for approval
  • Test responsive layout on phone + tablet emulator
  • Get visual sign-off before coding game logic

Phase 1 — Core Game Loop (4–6 weeks)

  • Android export configured, landscape locked
  • Responsive design system (anchors, font scaling, Camera2D zoom)
  • Touch input → quest dialog overlay
  • Placement quiz (assigns reading level 1–4)
  • 3 quest types: multiple choice, fill-in-blank, drag-and-drop
  • 6 building quest JSON files (3 difficulty levels each)
  • godot-sqlite local persistence, TTS read-aloud
  • XP bar, streak counter, HUD, unlock particles

Phase 2 — Cloud Sync + Teacher Dashboard (3–4 weeks)

  • Neon PostgreSQL schema + Cloudflare Worker REST API
  • Offline-first SQLite → Neon sync on WiFi
  • Teacher web dashboard (Cloudflare Pages)
  • Student login (name/PIN → Neon students table)
  • 12 buildings / full village map expansion

Phase 3 — Multiplayer + Store (4–6 weeks)

  • Reading Race (competitive real-time) + Squad Restore (cooperative)
  • Student Dashboard (Overview, Skill Detail, Multiplayer Stats)
  • Collaborative Room mode
  • Google Play Store submission (AAB, Target SDK 35)
